## 🐍 PYTHON — Variables, Types & The Dict

### Intuition

Python doesn't have *declarations*. When you write `x = 5`, you're not reserving memory for an integer — you're binding the name `x` to an object that happens to hold `5`. That object knows its own type. This is called **dynamic typing** and it's the single biggest conceptual gap for people coming from C/Java: the variable is just a label, not a box.

The type that will carry you furthest in this program is the **dict** (`{}`). It's Python's Swiss Army knife — hash map, namespace, JSON body, config store, and as you'll see in the DSA block, the key to turning an O(n²) solution into O(n). Learn to reach for it instinctively.

### The Essentials

```python
# --- Primitive types ---
# int, float, str, bool are immutable — reassigning creates a new object
name        = "Hasnat"      # str: sequence of Unicode code points
age         = 22            # int: arbitrary precision, no overflow
gpa         = 3.8           # float: IEEE 754 double precision
is_student  = True          # bool: subclass of int — True == 1, False == 0

# f-strings: the only string formatting you need to know
# The expression inside {} is evaluated at runtime, not parse time
print(f"Name: {name}, Age: {age}, GPA: {gpa:.2f}")
#                                         ^^^^ format spec: 2 decimal places

# --- Type conversion: explicit, never implicit ---
# Python will NOT silently coerce types across boundaries (unlike JS)
print(str(age) + " years old")   # int → str: safe concatenation
print(int("42"))                  # str → int: raises ValueError if not numeric
print(float("3.14"))              # str → float
print(bool(0), bool(""), bool([]))  # 0, empty string, empty list → all False

# --- Truthy / Falsy — memorise this table ---
# Falsy:  0, 0.0, "", b"", [], {}, set(), None, False
# Truthy: literally everything else, including "0" and [0]
# This matters: FastAPI uses bool(value) internally for optional fields

# --- The dict ---
# dict is a hash map: O(1) average lookup, insert, delete
# Key must be hashable (str, int, tuple of hashables) — list cannot be a key
scores = {"math": 95, "cs": 99, "physics": 88}

# Three ways to access — know all three
print(scores["cs"])              # KeyError if missing — use when key MUST exist
print(scores.get("bio", 0))      # returns default 0 if key absent — use defensively
print("math" in scores)          # membership test: O(1), not O(n) like a list

# dict.items() gives you (key, value) pairs — you'll use this constantly
for subject, mark in scores.items():
    print(f"{subject}: {mark}")

# dict comprehension — one-liner transformation
passing = {k: v for k, v in scores.items() if v >= 90}
# {'math': 95, 'cs': 99}
```

### Why it matters going forward

The dict is not just a data structure — it's a **pattern**. Whenever you catch yourself writing a second nested loop to "find" something you've already seen, that's a dict waiting to be written. You'll see this exact realisation play out in the next block.

## 🧩 DSA — Two Sum *(Easy · Arrays)*

**NeetCode:** https://neetcode.io/problems/two-sum

### The Problem

Given an integer array `nums` and a target `target`, return the **indices** `[i, j]` such that `nums[i] + nums[j] == target`. Exactly one solution exists. You may not use the same element twice.

### Reading the problem like an engineer

Before writing any code, reduce it to a question: *"For each number I'm looking at, do I already know its partner?"* The partner of `nums[i]` is `target - nums[i]`. If you could answer that question in O(1), you'd need only one pass through the array.

That's the whole insight. Everything below is just implementation.

### Approach 1 — Brute Force (write this first, always)

```python
def two_sum_brute(nums: list[int], target: int) -> list[int]:
    # Check every pair (i, j) where i < j
    # Starting j at i+1 avoids using the same element twice
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]
    return []
    # Time:  O(n²) — n*(n-1)/2 comparisons in the worst case
    # Space: O(1)  — no extra memory
    # Fine for n < 1000. Breaks at n = 10⁴ (10⁸ ops → seconds).
```

Never skip the brute force. It proves you understood the problem before you optimised it — interviewers notice.

### Approach 2 — Hash Map, One Pass (the insight)

```python
def two_sum(nums: list[int], target: int) -> list[int]:
    # seen maps each number we've visited to its index.
    # As we walk forward, we check: "is the complement already in seen?"
    # If yes — done. If no — add current number and keep walking.
    seen: dict[int, int] = {}   # value → index

    for i, num in enumerate(nums):
        complement = target - num   # what we NEED to pair with num

        if complement in seen:
            # Found it. seen[complement] is the earlier index, i is the current one.
            return [seen[complement], i]

        # Not found yet — record this number so future iterations can find it
        seen[num] = i

    return []  # guaranteed not reached (problem states exactly one solution)

    # Time:  O(n) — single pass, each element visited once
    # Space: O(n) — seen dict grows up to n entries in the worst case
```

### Walkthrough on `nums = [2, 7, 11, 15]`, `target = 9`

```
i=0  num=2   complement=7   seen={}        → miss → seen={2:0}
i=1  num=7   complement=2   seen={2:0}     → HIT  → return [0, 1]  ✓
```

The moment you find the complement, you stop. You never even touch indices 2 and 3.

### What if duplicates exist? `nums = [3, 3]`, `target = 6`

```
i=0  num=3   complement=3   seen={}        → miss → seen={3:0}
i=1  num=3   complement=3   seen={3:0}     → HIT  → return [0, 1]  ✓
```

Storing `seen[num] = i` *after* the check ensures you never match an element against itself — by the time `i=1` checks for `3`, only index `0` is in `seen`, which is a different element.

## ⚡ FASTAPI — Setup & Your First Route

### Intuition

FastAPI sits on top of Python's async web standard (ASGI) and does three things automatically that other frameworks make you do by hand:

1. **Validates** incoming data using Python type hints
2. **Serialises** return values to JSON
3. **Documents** every route at `/docs` (Swagger UI) — zero config

The mental model: write a normal Python function, decorate it with the HTTP method and path, return a dict. FastAPI handles the rest.

### Setup

```bash
pip install fastapi uvicorn[standard]
# fastapi    — the framework
# uvicorn    — ASGI server that actually listens for HTTP connections
# [standard] — pulls in websockets and httptools for performance
```

### Your first app

```python
# main.py
from fastapi import FastAPI

# FastAPI() creates the application instance.
# title and description appear in /docs — fill them in, you'll thank yourself later.
app = FastAPI(title="Sprint", description="Day 1 — Hello World")


# @app.get("/") registers a route: HTTP GET at the path "/".
# The function name becomes the operationId in the OpenAPI spec.
@app.get("/")
def root():
    # Returning a dict is all you need — FastAPI calls json.dumps() for you.
    return {"message": "Sprint is running.", "day": 1}


# Path parameters: curly braces in the path, matching parameter in the function.
# FastAPI reads the type hint `: str` and will coerce + validate automatically.
@app.get("/hello/{name}")
def greet(name: str):
    # If someone hits /hello/Hasnat, name == "Hasnat"
    return {"greeting": f"Assalamu Alaikum, {name}!"}
```

```bash
# --reload restarts the server whenever you save main.py — development only
uvicorn main:app --reload
```

Visit **http://localhost:8000/docs** — you get a fully interactive API explorer. Click any route, hit *Try it out*, send a real request. This is the killer feature: you never need Postman for development.

### What is Pydantic?

FastAPI uses Pydantic under the hood for validation. When you annotate a parameter with a type, FastAPI constructs a Pydantic field from it. If the incoming data doesn't match, FastAPI returns a `422 Unprocessable Entity` with a detailed error — before your function even runs.

```python
from pydantic import BaseModel

# BaseModel turns a plain Python class into a schema.
# Declare fields as class attributes with type hints.
# Pydantic validates types, coerces where safe, raises ValidationError otherwise.
class Item(BaseModel):
    name: str
    price: float
    in_stock: bool = True   # default value — field becomes optional in the request body
```

## 🔗 COMBINED TASK — Wiring All Three

Here's where the day pays off. The dict pattern from Python becomes the algorithm in DSA, which becomes the logic inside a FastAPI endpoint. Same idea, three contexts.

```python
# main.py — add this to your existing file
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI(title="Sprint", description="Day 1 — Hello World + Two Sum")


@app.get("/")
def root():
    return {"message": "Sprint is running.", "day": 1}


@app.get("/hello/{name}")
def greet(name: str):
    return {"greeting": f"Assalamu Alaikum, {name}!"}


# --- Two Sum endpoint ---

class TwoSumRequest(BaseModel):
    # List[int] tells Pydantic: this must be an array of integers.
    # Send anything else → FastAPI rejects with 422 before your code runs.
    nums: list[int]
    target: int


class TwoSumResponse(BaseModel):
    # Defining the response shape explicitly is good practice:
    # it appears in /docs and FastAPI filters out any extra fields you accidentally return.
    indices: list[int]
    values: list[int]
    found: bool


@app.post("/two-sum", response_model=TwoSumResponse)
def solve_two_sum(request: TwoSumRequest):
    """
    Accepts a JSON body: { "nums": [2, 7, 11, 15], "target": 9 }
    Returns:            { "indices": [0, 1], "values": [2, 7], "found": true }

    The algorithm is identical to the DSA block above — same dict, same one pass.
    The only new concept: the input arrives as a validated Pydantic model
    instead of raw function arguments.
    """
    seen: dict[int, int] = {}   # value → index (same as DSA block)

    for i, num in enumerate(request.nums):
        complement = request.target - num
        if complement in seen:
            j = seen[complement]
            return TwoSumResponse(
                indices=[j, i],
                values=[request.nums[j], num],
                found=True,
            )
        seen[num] = i

    # No solution found — return empty lists, found=False
    return TwoSumResponse(indices=[], values=[], found=False)
```

**Test it in Swagger (`/docs`):**

```json
POST /two-sum
{ "nums": [2, 7, 11, 15], "target": 9 }
```

Expected response:
```json
{ "indices": [0, 1], "values": [2, 7], "found": true }
```

Try sending `"nums": [1, "two", 3]` — watch FastAPI reject it with a 422 without you writing a single validation line.

## 🔑 CONCEPT ANSWER

**Question:** "Python's `dict` guarantees O(1) *average* lookup. What does *average* mean here, and can you construct a case where it degrades to O(n)?"

**Answer:** Python's `dict` is a hash table. Lookup works by computing `hash(key)`, mapping it to a bucket index, and reading the value — all O(1) if each key lands in a distinct bucket. *Average* means "averaged over many random keys under a uniform hash distribution." In the worst case, many keys can hash to the same bucket (**hash collision**), turning lookup into a linear scan of that bucket's chain — O(n) for n collisions. In practice, Python's hash function is designed so this almost never happens with real data. You *can* force it: before Python 3.6, all `str` objects with the same characters (in different order) would collide on certain CPython builds. Today, hash randomisation (enabled by default since Python 3.3 via `PYTHONHASHSEED`) makes deliberate collision attacks impractical. For interviews: say "O(1) amortised, O(n) worst case with adversarial keys, irrelevant in practice."
