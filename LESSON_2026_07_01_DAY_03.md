## 🐍 PYTHON — Lists: Slicing, Mutation & When a List Beats a Dict

### Intuition

A Python `list` is an ordered, **mutable** sequence — the opposite of the string immutability you learned on Day 2. `.append()`, `.pop()`, `.sort()`, and `.insert()` all change the list in place instead of returning a new one. That mutability is the entire reason lists exist: you build them up incrementally, which you can't do with a string.

Lists also support the same `[start:stop:step]` slicing as strings, because slicing isn't a string-only feature — it works on any sequence type.

### The Essentials

```python
nums = [5, 3, 8, 1, 9]

nums[0]     # 5   — first element
nums[-1]    # 9   — last element
nums[1:3]   # [3, 8]  — same slicing rules as Day 2's strings

nums.append(12)     # [5, 3, 8, 1, 9, 12]  — add to the end, O(1)
nums.pop()           # removes & returns the last item, O(1)
nums.pop(0)          # removes index 0, O(n) — everything shifts left
nums.insert(0, 100)  # insert at index 0, O(n) — everything shifts right

sorted(nums)   # returns a NEW sorted list, original untouched
nums.sort()    # sorts IN PLACE, returns None
```

### The membership question: list vs. set

```python
big_list = list(range(1000))
999 in big_list   # O(n) — has to scan, worst case, every element

big_set = set(big_list)
999 in big_set    # O(1) average — hash lookup, no scanning
```

This is the single most important performance fact about Python's built-in containers: **checking "is X in this collection?" is O(n) for a list but O(1) average for a set or dict.** Any time a problem is phrased as "have I seen this value before," reach for a set.

### Why it matters going forward

Today's DSA problem is exactly that question, asked directly: scan a list once, and the moment you see a repeat, stop. You'll use a `set()` the same way you used a `dict()` on Day 1 and Day 2 — as fast lookup memory of what's already been processed.

## 🧩 DSA — Contains Duplicate *(Easy · Arrays)*

**NeetCode:** https://neetcode.io/problems/contains-duplicate

### The Problem

Given an integer array `nums`, return `True` if any value appears at least twice, and `False` if every element is distinct.

### Reading the problem like an engineer

Strip the wrapper: *"As I walk through this array, have I already seen this exact value?"* That's a pure membership question — no counting, no frequency, just yes/no. Whenever a problem doesn't need "how many," don't reach for a dict; a set is the right tool and it says exactly what you mean.

### Approach 1 — Brute Force (write this first, always)

```python
def contains_duplicate_brute(nums):
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] == nums[j]:
                return True
    return False
    # Time:  O(n²) — every pair compared
    # Space: O(1)
```

### Approach 2 — Sorting

```python
def contains_duplicate_sort(nums):
    sorted_nums = sorted(nums)
    for i in range(1, len(sorted_nums)):
        if sorted_nums[i] == sorted_nums[i - 1]:
            return True
    return False
    # Time:  O(n log n) — dominated by the sort
    # Space: O(1) extra, ignoring what sorted() allocates internally
```

Sorting makes duplicates line up next to each other, so one adjacent-pair scan catches them. Better than brute force, but there's a linear-time option.

### Approach 3 — Hash Set (the insight)

```python
def contains_duplicate(nums):
    seen = set()
    for num in nums:
        if num in seen:
            return True
        seen.add(num)
    return False

    # Time:  O(n) — one pass, O(1) average membership check per element
    # Space: O(n) — worst case, every element is unique and goes into the set
```

### Walkthrough on `nums = [1, 2, 3, 1]`

```
seen = {}
num=1 -> not in seen -> seen={1}
num=2 -> not in seen -> seen={1,2}
num=3 -> not in seen -> seen={1,2,3}
num=1 -> 1 IS in seen -> return True immediately
```

Notice the early exit — the moment a duplicate shows up, you're done. Worst case (no duplicates at all) still costs one full pass, O(n).

### Why a set instead of a dict here?

A dict's extra value slot (the count) is wasted if all you ever check is "is the key present." A `set` is literally a dict with the values stripped out — same O(1) average lookup, less memory, and it reads as "a collection of things I've seen," which is precisely the intent.

## ⚡ FASTAPI — Request Bodies with Pydantic Models

### Intuition

Path and query parameters (Day 1, Day 2) are limited to simple values in the URL — strings, ints, bools. A **request body** lets a client send a full JSON object, which is what you need to POST a list, a nested object, or anything more complex than a single value. FastAPI uses a Pydantic `BaseModel` to describe the shape of that JSON and validates it automatically before your function ever runs.

### The Essentials

```python
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List

app = FastAPI(title="Sprint Day 3")


# A BaseModel subclass declares the shape of the expected JSON body.
# FastAPI parses the incoming JSON, validates every field's type,
# and hands you back a real Python object — not a raw dict.
class NumberList(BaseModel):
    nums: List[int]


@app.post("/check-duplicates")
def check_duplicates(payload: NumberList):
    return {"nums": payload.nums, "count": len(payload.nums)}

# POST /check-duplicates
# Body: { "nums": [1, 2, 3] }
# -> { "nums": [1, 2, 3], "count": 3 }
```

### How FastAPI decides: path, query, or body?

The rule chain, in order: a parameter named in the path string is a **path param**. A parameter typed as a plain `str`/`int`/`bool`/`float` that isn't in the path is a **query param**. A parameter typed as a Pydantic `BaseModel` is read from the **request body**. You never annotate which is which — FastAPI infers it from the type.

```python
class NumberList(BaseModel):
    nums: List[int]

@app.post("/check-duplicates")
def check_duplicates(payload: NumberList):
    ...
# `payload` is a BaseModel -> FastAPI reads it from the JSON body, not the URL
```

Send a malformed body — `{"nums": ["a", "b"]}` where `nums` should be ints — and FastAPI returns `422` automatically, before `check_duplicates()` runs at all. That validation is the entire point of using Pydantic instead of accepting a raw dict.

## 🔗 COMBINED TASK — Wiring All Three

The seen-set pattern from Python becomes the Contains Duplicate algorithm, which becomes the logic behind your first POST endpoint.

```python
# main.py — Day 3 combined endpoint

class NumberList(BaseModel):
    nums: List[int]


@app.post("/check-duplicates")
def check_duplicates(payload: NumberList):
    """
    POST /check-duplicates
    Body: { "nums": [1, 2, 3, 1] }
    -> { "nums": [1, 2, 3, 1], "has_duplicates": true }

    Identical algorithm to the DSA block — the only new concept is that
    nums arrives as a validated JSON body instead of a function argument.
    """
    seen = set()
    has_duplicates = False
    for num in payload.nums:
        if num in seen:
            has_duplicates = True
            break
        seen.add(num)
    return {"nums": payload.nums, "has_duplicates": has_duplicates}
```

```

```

Expected response:
```json

```

Then try `{ "nums": [1, 2, 3, 1] }` and watch `has_duplicates` flip to `true`.

## 🔑 CONCEPT ANSWER

**Question:** "Why is `list.pop(0)` considered a red flag in an interview, but `list.pop()` isn't?"

**Answer:** A Python list is backed by a contiguous array, so removing the *last* element (`.pop()`) is O(1) — nothing else moves. Removing the *first* element (`.pop(0)`) is O(n) — every remaining element has to shift left by one position to close the gap. Do that inside a loop and an algorithm that looks O(n) is secretly O(n²). If you find yourself repeatedly popping from the front of a list, that's the signal to reach for `collections.deque` instead, which supports O(1) pops from both ends.
